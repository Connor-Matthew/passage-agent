# AI 图文创作平台后端面试准备计划

## 目标

把这个项目从“看过源码”变成“能用自己的话讲清楚后端设计”。

不背 100 个问答，而是通过几条主线理解项目流程。因为面试问题角度很多，但项目本身是不变的。

## 简历定位

如果项目不是自己从零独立开发，不建议写“个人独立项目”。更稳妥的表达是：

> 基于 AI 图文创作项目进行后端源码学习与功能梳理，重点理解 FastAPI 接口设计、文章生成状态流转、SSE 流式输出、多策略配图和会员配额控制等模块，并完成本地部署与接口调试。

如果后续自己完成了功能修改或二次开发，再补充为“二次开发项目”。

## 总体学习方法

每次遇到面试问题，先判断它属于哪条主线，再用这条主线回答。

核心原则：

1. 不死背答案。
2. 不假装自己从零开发。
3. 先讲流程，再讲技术点。
4. 先讲为什么，再讲怎么做。
5. 每个回答控制在 40 秒到 1 分钟。

## 六条主线

### 1. 请求主线

回答“用户点了按钮后，后端发生了什么”。

```text
前端提交 topic
-> FastAPI 接收请求
-> 校验用户和配额
-> 创建 article 记录
-> 返回 taskId
-> 后台异步开始生成
```

能覆盖的问题：

- 创建文章接口怎么设计？
- 为什么要返回 taskId？
- 为什么不直接同步返回文章？
- 用户配额什么时候扣？

### 2. 状态主线

回答“任务怎么一步步往下走”。

```text
PENDING
-> TITLE_GENERATING
-> TITLE_SELECTING
-> OUTLINE_GENERATING
-> OUTLINE_EDITING
-> CONTENT_GENERATING
-> COMPLETED / FAILED
```

能覆盖的问题：

- 为什么要有 status 和 phase？
- 用户中途刷新怎么办？
- 服务重启后数据还在吗？
- 如何防止乱点按钮导致状态错乱？

### 3. Agent 主线

回答“AI 生成流程为什么要拆开”。

```text
Agent1 生成标题候选
-> 用户选标题
-> Agent2 生成大纲
-> 用户确认大纲
-> Agent3 生成正文
-> Agent4 分析配图需求
-> Agent5 生成图片
-> 合成完整图文
```

能覆盖的问题：

- 为什么不用一个大 Prompt？
- Agent 之间怎么传数据？
- 哪些地方用户可以介入？
- 失败后能不能局部重试？

### 4. SSE 主线

回答“前端为什么能实时看到生成过程”。

```text
前端用 taskId 建立 SSE 连接
-> 后端为 taskId 创建队列
-> Agent 生成时不断写消息
-> SSE 从队列取消息推给前端
-> 完成或失败后关闭连接
```

能覆盖的问题：

- 为什么用 SSE？
- 和 WebSocket 有什么区别？
- 长文本生成怎么流式返回？
- 连接断了怎么办？

### 5. 配图主线

回答“图片怎么来的”。

```text
正文生成
-> Agent4 分析哪里需要图
-> 得到 imageRequirements
-> 按 imageSource 选择图片策略
-> 调 Pexels / Mermaid / Iconify / Qwen Image 等
-> 上传腾讯云 COS
-> 失败则降级默认图
-> 插入正文
```

能覆盖的问题：

- 策略模式怎么用的？
- 多种图片来源怎么统一？
- 外部 API 失败怎么办？
- 为什么配图失败不能影响文章生成？

### 6. 商业化主线

回答“权限和钱是怎么管的”。

```text
用户创建任务
-> 判断普通用户 / VIP / admin
-> 普通用户扣配额
-> VIP 可使用高级配图
-> Stripe 处理支付
-> 支付成功更新用户身份
```

能覆盖的问题：

- VIP 权限怎么判断？
- 配额怎么扣？
- 为什么要事务？
- 支付失败怎么办？

## 第一阶段：先讲清楚整体流程

目标：不用技术细节，也能讲清楚用户从创建文章到生成完成的流程。

标准答案：

> 这个项目是一个 AI 图文创作平台。用户输入选题后，后端会创建一个文章任务，并生成唯一的 taskId。因为 AI 生成是长耗时任务，所以接口不会同步等待结果，而是先把任务保存到 MySQL，然后后台异步执行生成流程。整个流程分成标题、大纲、正文、配图几个阶段，每个阶段的结果都会落库。前端通过 SSE 连接实时接收生成进度，用户可以在标题和大纲阶段进行确认，最后后端把正文和图片合成为完整文章。

练习问题：

1. 用户点击“创建文章”后，后端做了哪几件事？
2. 为什么创建文章接口返回 taskId，而不是直接返回最终文章？
3. 为什么 AI 生成任务不能同步阻塞在 HTTP 请求里？

## 第二阶段：理解状态管理

目标：能讲清楚 `status` 和 `phase` 的区别。

要点：

- `status` 是任务整体状态：待处理、处理中、完成、失败。
- `phase` 是任务当前阶段：生成标题、等待选标题、生成大纲、等待改大纲、生成正文。
- 用户可以中途刷新页面，因为关键结果都保存到了 MySQL。
- 后端用阶段流转限制非法操作，比如还没生成标题时不能确认大纲。

练习问题：

1. 为什么要同时有 status 和 phase？
2. 用户生成标题后关闭页面，再回来还能继续吗？为什么？
3. 如何防止用户重复点击或跳过阶段？

## 第三阶段：理解 Agent 拆分

目标：能讲清楚为什么拆 Agent，不是为了炫技，而是为了控制流程。

要点：

- 一个大 Prompt 难以控制结构，也无法中间确认。
- 拆分后每个 Agent 只负责一个子任务。
- 中间结果可以落库，失败后更容易定位和重试。
- 用户可以在标题和大纲阶段介入。

练习问题：

1. 为什么不用一个大 Prompt 直接生成完整文章？
2. Agent 之间通过什么传递数据？
3. 拆 Agent 带来的代价是什么？

## 第四阶段：理解 SSE

目标：能讲清楚 SSE 解决的是“长任务进度实时返回”的问题。

要点：

- 文章生成是长耗时任务。
- 创建接口先返回 taskId。
- 前端用 taskId 建立 SSE 连接。
- 后端执行过程中不断推送 Agent 进度和流式文本。
- SSE 是服务端到客户端的单向推送，比 WebSocket 更简单。

练习问题：

1. 为什么用 SSE，不用普通 HTTP 轮询？
2. 为什么用 SSE，不用 WebSocket？
3. SSE 连接断开怎么办？

## 第五阶段：理解配图策略

目标：能讲清楚策略模式的价值。

要点：

- 图片来源很多，每种调用方式不同。
- 主流程不应该关心具体图片从哪里来。
- Agent4 只生成配图需求。
- 策略层根据 imageSource 选择具体服务。
- 最后统一返回 ImageResult。
- 失败时降级默认图，避免文章生成失败。

练习问题：

1. 为什么配图适合用策略模式？
2. 不同图片来源如何统一返回？
3. 为什么图片失败不能中断文章生成？

## 第六阶段：理解 VIP、配额和支付

目标：能讲清楚商业化闭环。

要点：

- 普通用户创建文章消耗配额。
- VIP 和管理员可以使用高级配图能力。
- 创建任务时校验权限和配额。
- 配额扣减需要事务，避免并发重复消费。
- Stripe 支付成功后更新用户会员状态。

练习问题：

1. 普通用户和 VIP 用户权限有什么区别？
2. 为什么扣配额要放在事务里？
3. 支付失败和图片失败的处理方式为什么不同？

## 面试诚实表达模板

当面试官问“这个项目是你自己做的吗？”时，可以这样回答：

> 这个项目最开始是我用来学习 AI 应用后端开发的项目，不是从零完全手写。我重点做的是本地部署、接口调试、源码梳理和后端核心流程理解。通过这个项目我系统学习了 FastAPI、任务状态流转、SSE 流式输出、多阶段 AI 生成和配额控制。现在我能讲清楚它的核心设计，也在基于它做二次开发。

## 后续补强方向

为了让项目更像自己的，建议至少完成一个真实二次开发功能：

1. 给文章生成增加“联网搜索”开关。
2. 给 SSE 增加搜索中、搜索完成、搜索失败事件。
3. 给任务执行增加更清晰的执行日志页面。
4. 给失败任务增加“从当前阶段重试”能力。
5. 给配额消费增加流水表。

优先推荐第 1 个或第 4 个，因为最容易在面试里讲出后端设计价值。
